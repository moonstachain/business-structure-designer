# 业务数据结构设计方法论

这份方法论用于把一个复杂业务拆成可理解、可配置、可复用的数据结构。

核心目标不是先“建表”，而是先回答三个问题：

1. 这个业务真正要管理的对象是什么。
2. 哪些信息属于决策入口，哪些只是实现细节或证据。
3. 仪表盘应该帮谁在什么时点做什么动作。

## 一、先分清业务的 5 个层次

推荐统一按下面 5 层拆解业务：

1. `阶段/场景层`
业务从哪里开始，到哪里结束，中间有哪些关键阶段。

2. `对象/能力层`
真正需要被管理、评估、选择、推进的对象是什么。
它可以是策略、模块、能力点、流程节点、业务单元。

3. `实现机制层`
这些对象通过什么机制落地。
它可以是流程字段、公式、自动化、权限、表单、脚本、外部系统。

4. `证据/素材层`
有没有截图、案例、逐字稿、SOP、附件、说明文档来证明“这个设计真的能用”。

5. `呈现/行动层`
管理者需要看什么，操作者需要点什么，诊断时要跳到哪里。

这个顺序很重要：

`先识别阶段 -> 再识别对象/能力 -> 再识别实现机制 -> 再挂证据 -> 最后设计仪表盘`

不要倒过来做。先堆字段、先做仪表盘，通常都会把结构做乱。

## 二、主表只放“决策对象”

主表的职责永远只有一个：

`给人一个清晰的决策入口`

主表适合放这些信息：

- 对象名称
- 对象 ID
- 摘要
- 所属阶段/场景
- 触发条件
- 预期结果
- 当前状态
- 优先级
- 负责人
- 关联机制或关联证据的索引

主表不适合直接承载这些内容：

- 原始逐字稿全文
- 大量附件对象 JSON
- 教学截图堆叠
- 实现细节长列表
- 多套互相竞争的口径
- 仪表盘专用衍生字段

判断标准很简单：

如果一个字段不能帮助用户快速回答“现在该选什么、做什么、跟谁有关”，它大概率不该进主表。

## 三、阶段表负责导航，不负责承载重数据

阶段/场景表的作用是：

- 给主表提供稳定导航
- 统一口径
- 让仪表盘有汇总轴

阶段表适合承载：

- 阶段名称
- 阶段 ID
- 阶段目标
- 进入条件
- 完成标准
- 关键指标
- 默认关注对象

阶段表不应该变成“另一个主表”。
如果一个阶段表里开始堆很多执行细节，说明对象层没有单独拆出来。

## 四、实现机制必须单独建模

很多业务设计失败，不是因为没有字段，而是因为把“对象”与“实现手段”混在一起了。

推荐单独抽一张 `实现机制表`，把这些信息沉淀下来：

- 机制名称
- 机制类型
- 适用对象
- 输入要求
- 输出结果
- 使用限制
- 自动化程度
- 维护成本
- 是否可复用

这样做的价值是：

- 主表保持简洁
- 可以横向比较同一对象的不同实现方式
- 仪表盘可以独立看“能力缺口”与“机制依赖”

## 五、证据表不是可有可无，它负责“教学”和“验证”

复杂业务里，光有设计说明不够，必须有证据层。

证据/案例/素材表适合承载：

- 案例标题
- 来源
- 截图或附件
- 原始记录
- 摘要
- 可借鉴点
- 关联对象

它有两个作用：

1. `教学作用`
让后来者知道这套结构怎么实际运行。

2. `验证作用`
防止主表只剩抽象口号，没有落地证明。

## 六、仪表盘只回答管理问题，不替代表结构

仪表盘不是“把表搬上墙”，而是回答具体问题。

建议分 3 层：

### 1. 总览层

回答：

- 当前业务在哪些阶段最重
- 哪些对象状态异常
- 哪些机制使用最集中
- 哪些关键指标偏离预期

### 2. 诊断层

回答：

- 某个阶段缺的是什么能力
- 某个对象依赖哪些机制
- 某个结果缺哪些证据
- 当前问题是结构问题、执行问题，还是素材缺失

### 3. 行动层

回答：

- 现在要跳转到哪条记录
- 要找哪个负责人
- 要补哪份证据
- 要调整哪个机制

仪表盘设计原则：

- 不直接堆字段
- 一个卡片只回答一个问题
- 每个卡片最好对应一个动作
- 汇总口径必须来自主表或阶段表，而不是临时拼接

## 七、如何判断一个业务该拆几张表

推荐用下面的判断法：

### 只需要 1 张表

当业务对象单一、阶段简单、实现机制固定、证据不重时。

### 需要 2-3 张表

当业务已经出现这些信号：

- 同一个对象有多个阶段
- 同一个对象依赖多种机制
- 有独立素材或案例需要挂接
- 不同角色关心的信息不一样

### 需要 4 张及以上表

当业务已经具备这些条件：

- 需要决策目录
- 需要阶段导航
- 需要独立机制管理
- 需要证据体系
- 需要仪表盘做多层观察

## 八、一个新业务从 0 到 1 的建模步骤

### Step 1. 先写业务问题

明确：

- 这个业务要优化什么
- 谁是主要使用者
- 什么算“看明白”
- 什么算“能执行”

### Step 2. 列出阶段

写出业务从入口到结果的阶段链。
阶段必须能被解释成“为什么要存在”，而不只是流程顺序。

### Step 3. 找出核心对象

问：

- 真正要被管理和选择的是什么
- 哪些记录是管理对象，哪些只是素材或说明

### Step 4. 分离实现机制

把“怎么做”从“要管什么”里拆出去。

### Step 5. 找证据

把 SOP、截图、案例、附件、逐字稿挂上去。

### Step 6. 最后才做仪表盘

先确定管理问题，再决定看板结构。

## 九、常见反模式

- 把主表做成杂物间
- 把附件和全文直接堆进主表
- 让阶段表承担对象管理职责
- 用仪表盘反向定义数据结构
- 把实现机制写死在一条长文本里
- 没有稳定 ID，只靠自然语言标题做主键
- 证据层缺失，导致后期难以复用和教学

## 十、默认推荐结构

对大多数复杂业务，优先用下面这套结构起步：

1. `业务主表`
2. `阶段/场景表`
3. `实现机制表`
4. `证据/案例表`
5. `仪表盘`

如果还有外部事实源，再额外补一张 `事实源表`，但不要把事实源和主表合并。
