# CRM 多维表 vs 专家策略库

这份文档把两个案例放在一起，对照出一套可迁移的业务建模方法。

## 一、两个案例的本质角色

### CRM 多维表

当前这张 CRM 表的真实结构是：

- 主字段：`初始化功能设置`
- 业务字段：`功能详情`
- 阶段字段：`流程阶段`
- 机制字段：`实现方法`
- 证据字段：`操作示例 / 了解更多 / 附件`

它不是业务流水表，而是：

`业务能力目录 + 实现手段映射 + 教学证据`

它解决的问题是：

- 如何把 CRM 系统拆成一组可讲解的能力单元
- 每个能力单元依赖哪些多维表能力
- 如何用样例和资料证明“这套设计可操作”

### 专家策略库

专家策略库已经被拆成 5 张表：

1. `策略主表`
2. `任务场景子表`
3. `GitHub仓库总表`
4. `Skill Pattern子表`
5. `素材/逐字稿子表`

它的抽象角色是：

`策略目录 + 事实源仓库 + Skill 模式 + 素材证据 + 场景入口`

它解决的问题是：

- 面对复杂任务时，先找哪类策略
- 这些策略的事实依据在哪里
- 哪些已经沉淀为 skill
- 有哪些素材可以反向验证或教学

## 二、结构对照表

| 设计层 | CRM 表 | 专家策略库 | 通用抽象 |
| --- | --- | --- | --- |
| 决策入口 | 初始化功能设置 | 策略主表 | 主表只放“要选什么、要管什么” |
| 导航层 | 流程阶段 | 任务场景子表 | 阶段/场景表负责分类和导航 |
| 实现层 | 实现方法 | Skill Pattern子表 | 实现机制要单独承接 |
| 事实源 | 隐含在业务需求和案例里 | GitHub仓库总表 | 事实源不能和主表混在一起 |
| 证据层 | 操作示例 / 了解更多 / 附件 | 素材/逐字稿子表 | 素材和案例单独存放 |
| 呈现层 | 工作台/仪表盘 | 策略总览与镜像导出 | 仪表盘只做观察和行动入口 |

## 三、CRM 表给出的关键启发

### 1. 一条记录不是“数据行”，而是“能力单元”

例如：

- 线索管理
- 商机阶段管理
- 销售权限隔离
- 仪表盘数据推送

这些都不是日常运营产生的事实记录，而是系统设计里的能力点。

这就是为什么这张表适合做：

`业务设计母表`

而不适合直接做：

`运营交易事实表`

### 2. 阶段字段让业务地图成立

`流程阶段` 把 14 条能力点串成一个完整业务图谱。
当前可识别的阶段包括：

- 线索管理
- 客户信息管理
- 商机管理
- 客户跟进记录管理
- 合同管理
- 销售驾驶舱
- 客户作战沙盘
- 销售团队管理

这说明阶段字段的真正作用不是“好看分类”，而是：

- 建立上下文
- 支持看板聚合
- 让能力点具备流程位置

### 3. 实现方法字段把“业务诉求”翻译成“可配置能力”

当前已识别的实现方法包括：

- 高级权限
- 公式字段
- 仪表盘功能
- 自动化流程
- 评论功能
- 人员字段
- 到期提醒
- 筛选查看
- 附件字段
- 表单功能
- 流程字段

这说明 CRM 表的高明之处在于：

它不是只说“要实现什么”，而是把“要实现什么”映射到了“靠哪些平台能力实现”。

## 四、专家策略库给出的关键启发

### 1. 主表必须和事实源分开

策略主表解决的是“用哪条策略”，不是“策略从哪来”。
事实源则单独放在 GitHub 仓库总表里。

这避免了两个常见问题：

- 主表太重
- 主表口径被技术细节污染

### 2. 证据层必须下沉

逐字稿、纪要、分享链接都不直接塞进策略主表，而是下沉到素材表。

这样做的价值是：

- 决策入口更干净
- 证据可以独立复用
- 教学和验证能力更强

### 3. Skill Pattern 让“实现层”有独立角色

策略不等于 skill。
一个策略可以还没沉淀成 skill。
skill 也不等于事实源仓库。

所以单独建 `Skill Pattern子表` 是必要的，它把“能否被自动化调用”从“策略是否成立”中拆开了。

## 五、两个案例共享的设计原则

### 原则 1. 主表永远只放决策对象

CRM 里是一条能力点。
策略库里是一条策略。

### 原则 2. 场景/阶段表负责导航

不要把阶段表做成另一个主表。

### 原则 3. 实现机制必须单独建模

无论是飞书功能、自动化、脚本能力，还是 skill 模式，都不应该直接糊进主表。

### 原则 4. 证据必须下沉

截图、附件、逐字稿、案例、原文，都应该成为证据层。

### 原则 5. 仪表盘不做事实源

仪表盘用来观察、诊断、行动，不用来定义业务对象。

## 六、可迁移的通用结构

如果你以后碰别的业务，优先用下面这套抽象：

1. `业务主表`
承载核心对象。

2. `阶段/场景表`
承载导航和分类。

3. `实现机制表`
承载“靠什么做成”。

4. `证据/案例表`
承载截图、案例、SOP、逐字稿、附件。

5. `仪表盘`
承载观察、诊断、行动入口。

6. `事实源表`
如果业务还有仓库、系统、外部平台或原始表，就单独补一张事实源表。

## 七、什么时候像 CRM，什么时候像策略库

### 更像 CRM 的情况

- 你在设计一个业务系统
- 你要做能力地图
- 你要说明“某能力靠哪些平台能力实现”
- 你需要教学和演示

### 更像策略库的情况

- 你在做复杂任务的决策入口
- 你要从多个来源挑选可执行方案
- 你要区分策略、事实源、实现模式、素材证据
- 你需要持续同步 GitHub、Skill、文档或素材

## 八、最终迁移方法

碰到新业务时，先问：

1. 我要管理的是“能力地图”还是“决策目录”？
2. 阶段/场景是否需要单独导航？
3. 实现机制是否足够复杂，值得单独成表？
4. 有没有证据层，还是所有信息都被塞进主表了？
5. 仪表盘展示的是“管理动作”，还是只是字段堆叠？

只要这 5 个问题答清楚，表结构通常就不会走偏。
