# 仪表盘信息架构建议

仪表盘不直接替代表结构，它只负责把结构化数据变成管理动作。

## 一、总览层

目标：

- 快速判断当前业务分布与异常

建议卡片：

1. `阶段分布`
- 每个阶段的对象数量
- 每个阶段的高优先级对象数

2. `对象状态分布`
- 候选、已验证、默认推荐、停用

3. `机制覆盖度`
- 每类机制被多少对象使用
- 哪些对象没有明确机制

4. `证据覆盖度`
- 哪些对象已有案例
- 哪些对象仍缺证据

## 二、诊断层

目标：

- 判断问题出在对象、阶段、机制还是证据

建议卡片：

1. `阶段缺口诊断`
- 某阶段对象少、状态差、证据不足

2. `机制依赖诊断`
- 某对象依赖过多机制
- 某关键机制缺可复用方案

3. `证据不足诊断`
- 对象没有案例
- 机制没有截图或 SOP

4. `责任归属诊断`
- 哪些对象无负责人
- 哪些阶段积压在同一责任人

## 三、行动层

目标：

- 让人从看板直接进入处理动作

建议卡片：

1. `待补对象`
- 点击进入业务主表记录

2. `待补机制`
- 点击进入实现机制表

3. `待补证据`
- 点击进入证据/案例表

4. `高优先级异常`
- 直接跳转到负责人或关联对象

## 四、设计原则

- 一个卡片只回答一个问题
- 每个卡片最好对应一个动作
- 汇总口径优先来自主表和阶段表
- 机制与证据只在诊断层深入展开
- 不直接展示原始全文或附件 JSON
- 用户看完某张卡片后，应明确下一步跳到哪张表
